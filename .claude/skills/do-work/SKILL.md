# Do Work

Execute a complete unit of work: plan it, build it, validate it, commit it.

## Workflow

### 1. Understand the task

Work usually arrives as a GitHub issue on `rupertgermann/open-recall` labeled `ready-for-agent`. If no issue was named, list candidates:

```
gh issue list -R rupertgermann/open-recall --label ready-for-agent --state open
```

- Issues come in two kinds: a **PRD** (parent, describes the whole feature) and **slices** (vertical tracer-bullet units). Implement one slice per unit of work.
- Read the slice's "Blocked by" section — implement blockers first; confirm they are closed or already merged before starting.
- Read the parent PRD for context, but scope the work to the slice only.
- PRDs intentionally avoid file paths and code snippets — locate the real code yourself; don't assume the issue text matches current file layout.

Read `CONTEXT.md` (project glossary). Use its domain language — Document, Chunk, Entity, Entity Key, Derived Document Data, Document Persistence, etc. — in code, tests, and commit messages. Respect its "Avoid" terms.

Explore the codebase to understand the relevant files, patterns, and conventions. If the task is ambiguous, ask the user to clarify scope before proceeding.

### 2. Plan the implementation (optional)

If the task has not already been planned, create a plan for it.

### 3. Implement

**For backend code**: use red/green/refactor, one test at a time in a tracer-bullet style.

1. Write a single failing test for the smallest vertical slice of behavior
2. Run the test — confirm it fails (red)
3. Write the minimum code to make it pass (green)
4. Repeat from step 1 for the next slice of behavior
5. Refactor if needed while keeping tests green

Each test should target one thin vertical slice through the system. Do not write all tests upfront — write one, make it pass, then move to the next.

Tests live in `tests/*.test.ts` and run on the built-in Node test runner (`node --test --experimental-strip-types`). Follow the style of the existing test files.

**For frontend code**: implement directly without TDD.

### 4. Validate

Run the feedback loops and fix any issues. Repeat until both pass cleanly.

```
npx tsc --noEmit
npm test
```

There is no `typecheck` npm script — use `npx tsc --noEmit` directly. Ignore errors under `.next/` (generated by the Next canary build, pre-existing); only errors in `src/` and `tests/` count as failures.

Optionally run `npm run lint` when the change touches many files.

### 5. Commit

Never commit on `main` or release branches. If on `main`, create a feature branch first (e.g. `feat/<slice-slug>`).

Once typecheck and tests pass, commit with a clear, descriptive message in the repo's conventional style (`feat(scope): …`, `fix(scope): …`, `docs: …`). Reference the issue number (e.g. `Closes #27`) so the slice closes on merge.
